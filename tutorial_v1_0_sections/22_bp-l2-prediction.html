Layer 2 - Prediction, Fused View, And Revision Barriers

Layer 2 uses more complex prediction than Layer 1 because it coordinates two inventory components. The UI must display pending operations without losing consistency with server state.

Operation Events

EventUse
OnInteractionOperationPredictedShow an immediate pending overlay.
OnInteractionOperationConfirmedRemove overlay and show confirmation feedback.
OnInteractionOperationRejectedShow error and refresh from authoritative view.
OnInteractionOperationCorrectedShow server correction and realign UI.
OnInteractionOperationInvalidatedOperation cancelled by session change or invalid state.

Pending Operation Views

Use GetPendingOperationViews for debug UI and to show how many operations are waiting.

Pending Debug
PendingViews = Interaction.GetPendingOperationViews()
ForEach PendingView:
  Print OperationId, OperationType, Resolution, bWaitingForBarrier

Revision Barrier

A Layer 2 operation can be confirmed by the server but finalized by the UI only after the client has observed the updated revisions of the involved endpoints.

  • CrossEndpoints: primary and secondary must reach the required revision.
  • SecondaryOnly: only the secondary endpoint is required.

In most cases, you do not need to manage the barrier manually, but you should understand why an operation can remain pending for a few frames after the ack.

Fused View

GetSecondaryPredictedView combines authoritative state and predicted overlay. Use it to draw the secondary endpoint during pending operations.

FieldMeaning
PendingSecondaryOverlayCountNumber of active predicted overlays.
BaseSecondaryReplicatedRevisionBase authoritative revision.
RootContainersContainers to draw in UI.
bHasSyntheticPredictedItemsThere are items with temporary client-side handles.

Synthetic Handles

When prediction creates an item on the client before the server assigns the final handle, the UI may see a synthetic handle. Do not save it permanently. Always rely on refresh and remapping handled by the system.

Latency Test

  1. Simulate high latency.
  2. Deposit a stack from the player to the chest.
  3. The secondary panel should update immediately through fused view.
  4. During pending state, the status bar shows pending count.
  5. After confirmation, synthetic and pending overlays disappear.

Verification

  • The UI remains responsive even with latency.
  • Rejects and corrections do not leave ghost items.
  • Fused view is used only for the secondary endpoint during an active session.
