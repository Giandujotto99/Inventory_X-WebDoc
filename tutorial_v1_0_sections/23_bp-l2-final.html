Layer 2 - Complete Multiplayer Chest Checklist

This checklist closes the Blueprint section. If every point is complete, the user has an autonomous foundation for inventories, stashes, loot, and multiplayer providers.

Functional Checklist

AreaCompleted when...
PlayerControllerIt has exactly one valid InventoryInteractionComponent.
Primary inventoryThe Layer 1 player inventory works and is resolved by the Layer 2 component.
ProviderThe chest exposes ProvidedInventoryComponent, anchor, tags, and policy.
PolicyRead/write, fallback, capabilities, range, and lifecycle are configured.
SessionsOpen, close, resume, upgrade, and downgrade work.
Dual UIPrimary and secondary are displayed correctly.
Cross operationsTransfer, split, merge, and swap between endpoints are wired.
Secondary operationsMove, split, merge, swap, and rotate inside the chest respect capabilities.
SecurityTwo players cannot write to the same chest at the same time.
PredictionPending, fused view, confirmed, rejected, and corrected states are handled in UI.

Minimum Tests Before Shipping

  1. Single player: open chest, deposit, withdraw, close.
  2. Listen server: client deposits and server sees correct replication.
  3. Dedicated server: client opens session, operates, and receives ack.
  4. Two clients: lock and read-only fallback work.
  5. High latency: UI remains coherent with fused view.
  6. Range loss: session is suspended or closed according to policy.
  7. Forced reject: UI recovers without ghost items.

Common Problems

SymptomCheck
Open session failsNull provider, missing policy, unresolved primary inventory, or wrong player controller.
Read-only session when you expected writeWrite lock already taken, fallback active, or policy does not grant write.
Operation returns CapabilityDeniedThe specific capability is missing from the read/write bitmask.
Operation returns WriteLockRequiredThe session does not hold write lock.
Transfer fails with inventory resultLayer 1 error: space, placement, stack, equip, or rule.
Secondary UI does not updateYou are not using GetSecondaryPredictedView or you did not bind events.

Natural Extensions

  • Vendor inventory with withdraw/deposit capabilities controlled by quest state.
  • Corpse loot read-only for some players and write for owner or party.
  • Vehicle cargo with wider range and line of sight disabled.
  • Guild stash with requester tags and primary owner tags.
  • Mobile UI with simplified drag and transfer buttons.

From here, you can treat Inventory_X as a reliable backend and build your specific UX on top of it: RPG, extraction shooter, survival, tactical inventory, or persistent loot system.
