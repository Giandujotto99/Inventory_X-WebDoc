Layer 1 - Multiplayer And Client Prediction

Inventory_X is server-authoritative but applies prediction on the owning client. This tutorial explains how to test it and how to build a UI that handles pending, confirmed, rejected, and corrected operations.

PIE Setup

  • Play As Client with 2 players, or Listen Server + Client.
  • For advanced testing, use a Dedicated Server.
  • Enable latency simulation from the editor or console net emulation if available.
  • Make sure the character replicates and the InventoryComponent is on the correct player owner.

Prediction Events

EventWhen to use it
OnPredictedOperationAppliedThe client has immediately applied a local operation.
OnOperationConfirmedThe server confirmed the predicted operation.
OnOperationRejectedThe server rejected the operation.
OnOperationCorrectedThe server corrected the predicted state.
OnPredictedOperationInvalidatedA prediction is no longer valid because of authoritative state.
OnAuthoritativeRevisionObservedThe client received a new authoritative revision.

Pending UI

When a command wrapper returns Status = Pending, mark the item or cell as pending. Do not block the entire UI, but avoid sending many incompatible operations on the same item.

Pending Tracking
Result = Inventory.TryMoveItem(...)
If Result.Status == Pending:
  PendingOperationIds.Add(Result.OperationId)
  Add visual overlay to affected item

OnOperationConfirmed(OperationId):
  Remove pending overlay

OnOperationRejected(OperationId, ResultCode):
  Remove pending overlay
  Show error feedback

Confirmation Test

  1. Start 2 players with simulated latency.
  2. Move an item inside the backpack.
  3. The client should see the move immediately.
  4. After the server ack, the pending overlay disappears.

Reject Test

To generate a reject, create a case where the client tries a move that is no longer valid for the server. For example, with two windows or server debug logic, occupy the target cell before the client receives replication.

Expected result: the client receives OnOperationRejected or a correction, and the UI realigns with authoritative views.

Rules To Follow

  • Never save parallel inventory state in the UI.
  • Do not treat Pending as final success.
  • Always use operation ids to connect UI feedback and server acknowledgments.
  • After reject or correction, call RefreshInventory and read views from the component.

Verification

  • Owning client operations feel immediate.
  • Other clients only see the state replicated by the server.
  • The UI recovers correctly after reject or correction.
