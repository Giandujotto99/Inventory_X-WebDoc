Layer 2 - Security, Locks, And Real Multiplayer Cases

This part prevents exploits, race conditions, and conflicts when multiple players interact with the same external inventory.

Two-Player Test

  1. Start PIE with 2 clients.
  2. Both clients look at the same BP_WorldChest.
  3. Client 1 opens a write-exclusive session.
  4. Client 2 tries to open a write-exclusive session.
  5. If fallback is enabled, Client 2 enters read-only. If fallback is disabled, Client 2 receives a reject.

Lock State

Lock StateUI meaning
NoneNo valid session or uninitialized state.
UnlockedThe session does not hold an exclusive lock.
HeldByThisSessionThis player can execute mutating operations if the capability is granted.
HeldByOtherSessionAnother player has the lock. Show read-only state or a disabled request upgrade button.

Capability Denied

If an action is not granted by the policy, the result can be CapabilityDenied. The UI should prevent the click, but you still need to handle the reject.

Range And Visibility

With bRequireInteractionRange enabled, the provider checks the distance between requester and anchor. If the player moves away, the session can be suspended.

  • OutOfRange: access denied or session invalid because of distance.
  • SessionSuspended: the session exists but cannot execute normal operations.
  • OutOfRangeGrace: tolerance period before suspension.

UI During Suspended State

When you receive OnInteractionSessionSuspended:

  • Show a Session suspended overlay.
  • Disable mutating operations.
  • Show the reason, for example out of range or visibility lost.
  • If the ResumeSuspendedSession capability is present, show a Resume button.

Upgrade And Downgrade

A read-only player can request write lock if:

  • The policy allows bAllowWriteLockUpgrade.
  • The UpgradeToWriteLock capability is granted.
  • No other player already owns write lock.

Downgrade is useful when you want to keep viewing the chest but release the lock for other players.

Verification

  • Two players cannot write to the same provider at the same time.
  • Read-only fallback works if enabled.
  • Range loss suspends or closes the session according to policy.
  • The UI clearly explains why an action is not available.
