Layer 1 - Blueprint Architecture

Before creating the inventory, you need to understand what you are wiring in Blueprints. Layer 1 manages one inventory: containers, items, fragments, rules, layout, networking, and prediction.

Main Concepts

Concept Practical meaning
UInventoryComponent The runtime component to add to a character, controller, chest, or any actor that owns an inventory.
Container A space that contains items, do not get confused: the container is not the inventory, an inventory can have multiple containers. It can be grid-based, like an NxM backpack, or slot-based, like equipment slots.
Item A runtime instance created from an InventoryItemDefinition.
Fragment Modular data attached to an item: ammo, durability, condition, or custom data.
Handle A safe reference to an item, container, fragment, or operation. Do not save array indexes instead of handles.
View A read-only snapshot for Blueprint UI and queries. Views are the correct way to read state.
Operation A change request: add, move, swap, split, merge, rotate, equip, or unequip.

Definition vs Runtime

Data Assets describe what can exist. The component contains what actually exists during gameplay.

Mental Model
InventoryItemDefinition = item type, for example Rifle
InventoryItemRecord     = runtime Rifle instance in the player inventory
InventoryItemView       = Blueprint snapshot used to read that instance
InventoryItemHandle     = stable reference to that instance

Operation Flow

When you call a node such as TryMoveItem, you are not simply moving a widget. You are submitting a request to the inventory system.

1

Blueprint calls a command wrapper

Examples: TryMoveItem, TryAutoAddItem, TryEquipItem.

2

Inventory_X creates an operation request

The request contains operation type, item handle, container handle, placement, and quantity.

3

The Rules Engine validates it

It checks handles, permissions, grid space, stacking, equipment compatibility, nesting, and rotation.

4

The Operation Engine applies the change

If the request is valid, it updates records, layout, fragments, and replication state.

5

Delegates notify UI and gameplay

Events such as OnInventoryChanged or OnItemUpdated let the UI refresh.

Authority And Prediction

Inventory_X is server-authoritative. In multiplayer, the server decides whether an operation is valid. On the owning client, however, many Layer 1 operations are simulated immediately to avoid input lag.

  • On Authority: the operation is validated and applied immediately.
  • On Autonomous Proxy: the operation is predicted locally and sent to the server.
  • On Simulated Proxy: the operation is rejected with AccessDenied.
Practical rule

The UI should always read from component views, not from custom parallel variables. If you keep duplicated inventory state in the UI, it will eventually go out of sync.
