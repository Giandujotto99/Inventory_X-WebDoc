Cross-Endpoint Validation



Cross-Endpoint Validation (Phase 5) represents the logical layer of Layer 2 responsible for verifying the semantic validity of a mutation between two distinct inventories. While the security system (Phase 4) authorizes the action based on identity and permissions, the Rules Engine determines whether the operation makes sense on the geometric, stacking, equipment, and nesting levels.

The Validation Context

The system operates within an FInventoryInteractionRuleContext, a unified read-only context that aggregates session data and Layer 1 contexts for both endpoints.





Bridge to Layer 1: Through the Layer1Bridge, the system derives local authority contexts for the Primary and Secondary, allowing the reuse of mature single-inventory rules without duplicating them.



Determinism: Every operation follows a fixed validation order and interrupts the chain at the first failure (early return principle).

1. Scoped Handles Verification

This is the first validation gate. The system ensures that every reference to objects or containers is consistent with the actual session endpoints:





Verifies that the handles actually belong to the declared role (Primary or Secondary).



Checks that the handles are resolvable in their respective destination components.



Validates the request structure (e.g., a TransferItem must have opposite source and destination endpoints).

2. Distributed Reuse of Layer 1

Layer 2 delegates “local” checks directly to Layer 1, then maps the outcome to the specific inventory involved:





Local Access: Verifies whether the individual containers involved have local restrictions (e.g., a container in the chest marked as ReadOnly at Layer 1 level).



Acceptance (Queries): Checks whether the destination container accepts the item through tag queries.



Geometry and Rotation: For grid placements, it queries the geometric engine of the target component to verify free space and collisions.

3. Cross-Endpoint Semantic Rules

Some checks require specific logic that crosses the boundary between the two components:

Cross-Stacking (Merge and Split)





Merge: Unlike Layer 1, merging between two different components requires a deterministic comparison of fragments (durability, ammunition, etc.), ordering them by FragmentTypeTag to ensure structural equivalence.



Enterprise Restrictions: In v1, stacking operations (split/merge) are prohibited if the objects have child containers, to avoid ambiguities in the data state.



Full Transfer: TransferItem must always involve the entire source stack; partial movements must explicitly go through SplitStack.

Cross-Nesting (Depth)

The system validates the resulting depth of the subtree being moved. The formula ensures that the union of the source hierarchy with the destination hierarchy does not exceed the global MaxNestingDepth limit (set to 3).

4. The Complex Case: SwapItems

The exchange of two items between different inventories is handled as two symmetrical “legs” but validated in deterministic order:





The Source -> Target Location link is validated (using the position occupied by the second item as the landing zone).



The Target -> Source Location link is validated.



Geometric Exclusion: During grid checking, the system “ignores” the item that is about to leave the destination cell, preventing false “cell occupied” errors during swap simulation.

Layered Outcome

The Rules Engine does not return a simple boolean, but a result that clearly distinguishes the origin of the error:





Layer 2 Failure: Orchestration errors (e.g., InvalidScopedHandle, InvalidEndpoint).



Primary/Secondary Failure: Local Layer 1 errors (e.g., CellOccupied or OutOfBounds) correctly mapped to the responsible endpoint.
