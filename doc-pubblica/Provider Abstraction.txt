Provider Abstraction

The Provider Abstraction is the Layer 2 mechanism that allows Inventory_X to interact with any entity in the world (chests, corpses, lockers, vehicles) without needing to know the specific class of the object.
Instead of relying on hardcoded logic, the system uses an interface and an adapter component to canonicalize every external inventory.
1. The Provider Interface (IInventoryExternalInventoryProvider)
This interface represents the minimum contract that Layer 2 expects from an external inventory source. Any Actor that implements it must be able to provide the system with:
The reference to the real UInventoryComponent that hosts the items.
An Interaction Anchor (a component or a physical point) to calculate distance and visibility.
The associated Interaction Policy asset to regulate permissions.
Any classification Tags and a readable name for debugging.
2. Adapter Component (UInventoryExternalInventoryProviderComponent)
To facilitate integration, the plugin provides a ready-to-use component that implements the interface as a convenience adapter.
This allows transforming an existing Actor into a provider simply by adding the component and configuring the references via the Editor, without having to write C++ code or implement interfaces in Blueprints.
The component supports intelligent fallbacks to minimize configuration:
If the interaction anchor is not set, it uses the Actor’s root.
If the inventory is not explicitly linked and the Actor has exactly one UInventoryComponent, the system selects it automatically.
3. Resolution Rules (Precedence)
When a player attempts to open a session toward an Actor, the internal Resolver applies a fixed and unambiguous precedence logic:
It checks if the Actor itself implements the interface.
If not, it searches among the Actor’s components for one that implements the interface.
If it finds exactly one valid component, it uses it as the provider.
If it finds nothing or finds multiple valid components (ambiguous situation), the resolution fails safely to avoid logical errors.
4. Canonical Identity
A cornerstone of the architecture is that the session’s "Secondary Endpoint" is not the physical Actor, but the UInventoryComponent resolved through the provider.
This ensures that the Writer Lock and Revisions are applied to the real data and not to simple visual containers.
