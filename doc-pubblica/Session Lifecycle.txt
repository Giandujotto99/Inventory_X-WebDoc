Session Lifecycle

The Session Lifecycle is the mechanism that governs the existence and validity of the link between the player and an external inventory (such as a chest or a storage). Unlike Layer 1, where access is immediate, in Layer 2 every interaction must pass through a series of logical states authorized by the server to ensure security and synchronization.

Here are the fundamental details of the enterprise lifecycle:

1. Session States

The system models the lifecycle through explicit states that define what is allowed at a given moment:

Opening: Transition phase in which the server validates the request, resolves the provider, and decides the initial mode.

ActiveReadOnly: The session is active. The player can see the content but does not hold the writer lock to modify it.

ActiveWriteExclusive: The session has full control. Only this player can perform mutations (moves, split, merge) on the external inventory.

Suspended: A transitional “protection” state. The session remains alive, but active operations are blocked while waiting for conditions to be restored (e.g., returning within range).

Closing / Closed: The session is terminated, locks are released, and resources are deallocated.

2. The Concept of SessionEpoch (Logical Generation)

To prevent conflicts or the use of obsolete requests, each session is marked by a SessionEpoch. This counter increases every time the session’s “operational boundary” changes significantly:

Transition from Read-Only to Write (Upgrade) or vice versa.

Suspension and subsequent restore (Resume).

Substantial change in permissions (Capability Snapshot).

Any operation built by the client during a previous epoch is automatically invalidated if it reaches the server when the epoch has already changed, ensuring total transaction integrity.

3. Writer Lock Dynamics

The system follows a Multi-View / Single-Writer model. While many players can be in ActiveReadOnly on the same chest, the server grants the ActiveWriteExclusive state to only one user at a time.

Upgrade/Downgrade: A player can request to switch to write mode. If the lock is occupied, the server can deny the upgrade or offer a read-only fallback.

No Forced Stealing: In the current version, a user cannot “steal” the lock from another. The release must be voluntary, due to timeout, or due to disconnection of the current owner.

4. Suspension and Resume (Grace Window)

The Suspended state is an advanced feature that manages recoverable interruptions. If a player leaves the range or loses line of sight, the server does not immediately close the session but suspends it.

During suspension, the writer lock is released to allow others to interact.

The player can request Resume as soon as conditions become valid again, obtaining a new epoch and resuming interaction without having to reopen the entire UI panel.

5. Session Termination

The session is invalidated and permanently closed in the presence of critical events:

Player disconnection or Pawn destruction.

Removal or destruction of the external inventory from the world.

Expiration of the “grace window” during a suspension.

Explicit closure of the interface by the user.
