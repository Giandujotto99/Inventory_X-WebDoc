Command Wrapper

The Command Wrappers represent the public operational surface of Inventory_X, designed to offer a smooth and ergonomic development experience both in C++ and in Blueprint. These wrappers act as high-level functions that hide the complexity of manually constructing requests and handling network protocols.

Here are the fundamental details of the Command Wrappers divided by the two levels of the system.

Design Philosophy

Ergonomics and Safety: Instead of requiring the user to manually configure complex data structures such as FInventoryOperationRequest, the system exposes typed functions (e.g., TryMoveItem) with clear and readable parameters.

Path Unification: The Blueprint APIs call exactly the same C++ code used internally by the plugin. There is no “separate backend” for Blueprints, ensuring that the validation rules are identical in every context.

Transparent Networking and Prediction: When a wrapper is invoked, the system automatically detects the Actor’s role (Authority, Autonomous Proxy, or Simulated Proxy). If necessary, it forwards the request to the server via RPC and instantly activates Client Prediction to eliminate latency perceived by the player.

Layered Verdicts: The functions do not return a simple boolean, but result structures that describe the outcome of the operation, the transaction state (e.g., Pending or Confirmed), and the handles of the objects involved.

--------------------------------------------------------------------------------

LAYER 1 Wrappers (Single Inventory Logic)

These functions, exposed by UInventoryComponent, manage internal mutations within the same inventory:

TryAddItem: Creates a new object starting from a definition in a specific position.

TryAutoAddItem: A convenience function that automatically finds the first free space suitable for the object.

TryRemoveItem: Destroys a record from the inventory, cleaning up the related data.

TryMoveItem: Moves an object between containers or different cells within the same component.

TrySwapItems: Swaps the position of two existing objects.

TryRotateItem: Rotates an object by 90 degrees if geometry and rules allow it.

TrySplitStack / TryMergeStack: Manages splitting a group of objects or merging two compatible stacks.

TryEquipItem / TryUnequipItem: Manages the transition between grid containers and typed equipment slots.

--------------------------------------------------------------------------------

LAYER 2 Wrappers (Cross-Inventory Interaction)

These functions, exposed by UInventoryInteractionComponent, coordinate atomic transactions between two distinct inventories (e.g., Player and Chest):

TryTransferItem: Moves an entire object (or subtree) from one endpoint to the other.

TrySplitStack (Cross): Splits a stack in one inventory and creates the result in the other.

TryMergeStack (Cross): Merges a stack belonging to one endpoint into a target stack located in the other.

TrySwapItems (Cross): Swaps two objects located in different inventories, guaranteeing the atomicity of the action (both move or neither does).

--------------------------------------------------------------------------------

Note for multiplayer integration: While Layer 1 operations provide immediate feedback via prediction, Layer 2 operations follow a barrier-driven model: the action is considered confirmed only when the client has observed the updated revisions of both involved inventories.
