Client Prediction

To eliminate the perception of network latency and ensure a smooth user experience, Inventory_X implements an advanced Client Prediction (Predictive Simulation) system. This technology allows the player to immediately see the outcome of their actions, while the server silently validates and confirms the operation in the background.

Here is how the plugin manages responsiveness in multiplayer contexts:

1. Instant Responsiveness (Immediate Feedback)

When the client owner (the Autonomous Proxy) sends a request to the server, the system does not wait for the network response. Instead, it instantly performs a local simulation of the operation, updating the UI and the visible state of the inventory as if the action had already been approved.

Local Validation: Before sending the request to the server, the client performs a preliminary check based on the authoritative data in its possession; if the action is clearly invalid (e.g., an item too large for the available space), the simulation is not started and the request is not sent, saving bandwidth.

2. Reconciliation Cycle (Reconciliation)

The system uses the state replicated by the server as the single source of truth. Every time new authoritative data or confirmations from the server (Ack) arrive, the client starts a Reconciliation Protocol to harmonize the local view:

Suspension: The simulated view is temporarily “frozen”.

Rollback: The effects of local predictions are undone to return to the pure authoritative state.

Alignment: The real data received from the server is applied.

Replay: All pending operations (not yet confirmed by the server) are “replayed” on top of the new authoritative state to rebuild the optimistic view.

3. Outcome Management and Corrections

Each predicted action ends with one of the following final outcomes:

Confirmed: The prediction was correct and the server confirmed the action without discrepancies.

Rejected: The server rejected the action; the system cancels the prediction and returns the inventory to the previous state (Rollback), updating the player’s UI.

Corrected: The action is valid, but the server’s final result differs slightly from the simulation (e.g., the item was placed in a different cell or received a different identifier). The system smoothly corrects the local view to align it with the server.

Invalidated: A pending operation is automatically canceled if it depended on a previous action that was rejected by the server (cascading invalidation).

4. Identity Management (Handle Remapping)

To handle complex operations, such as creating new objects (Split Stack or Add Item), the system uses a Handle Remapping Table. If the client generates a temporary identifier for a new object and the server assigns a different one, the system automatically updates all references in subsequent operations still in the queue, ensuring that the chain of actions remains consistent.

5. Security and Timeout

To prevent the UI from remaining stuck in a “predicted” state due to connection issues, the system implements a Security Timeout. If the server does not respond within a predefined time (e.g., 10 seconds), the operation is marked as failed and the client forces a realignment to the authoritative state.
