Instance Fragments

To manage dynamic data that changes during gameplay (such as weapon durability, the number of bullets in a magazine, or specific modifiers), Inventory_X uses the Instance Fragments system.

This approach allows item records to remain lightweight and performant, adding complex data blocks only when strictly necessary.

Vision and Philosophy

Fragments are modeled as pure data (not heavy objects) and are optional: the definition of an item decides which fragments to instantiate initially. Thanks to the use of FInstancedStruct, the system supports heterogeneous data in an extensible way, allowing developers to create new fragment types in C++ without having to modify the core of the plugin.

1. Fragment Definition (UInventoryItemFragmentDefinition)

Each fragment originates from a definition asset that acts as a template. The main fields of the asset include:

Fragment Type Tag: A unique GameplayTag (e.g., InventoryX.Fragment.Durability) that identifies the fragment logic.

Default Fragment Data: An FInstancedStruct container that hosts the initial data (e.g., maximum and current durability values).

2. Runtime State and Registry

At runtime, fragments are stored in a dedicated Flat Registry within the UInventoryComponent. The runtime record (FInventoryFragmentRecord) contains:

Fragment Handle: A stable reference to the fragment instance.

Owner Item Handle: The link to the item the fragment belongs to.

Fragment Data: The actual data instance, duplicated from the definition at creation time to ensure that runtime changes do not affect the original asset.

3. Key Features

Tag Contribution: Fragments can contribute Runtime Tags to the owning item. For example, a durability fragment could add the tag InventoryX.State.Condition.Broken to the item when the value drops to zero.

Delta Replication: Fragment data is fully supported in multiplayer through delta replication, ensuring that only data changes (e.g., the loss of one durability point) are transmitted over the network.

Canonical Comparison: During merge operations (stack merging), the system compares fragments following a canonical order based on type tags to ensure that identical items are merged correctly regardless of the fragment creation order.

4. Included Examples

The plugin provides some common fragment structures as reference:

FInventoryDurabilityFragmentData: Integrity and wear management.

FInventoryAmmoFragmentData: Loaded ammo count.

FInventoryConditionFragmentData: Preservation states or alterations.

5. Access via Blueprint

To facilitate use by technical designers, the system exposes typed helpers that avoid direct manipulation of raw structs:

TryGetDurabilityFragment

TryGetAmmoFragment

HasFragment (for quick queries about the presence of a capability)
