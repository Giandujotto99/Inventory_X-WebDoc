Layer 1 and Layer 2 - When To Use What

To understand which module to integrate into your workflow, it is essential to distinguish between managing the data of a single inventory and secure coordination between two different actors. Inventory_X is designed in a modular way: Layer 1 is the technological core that is always necessary, while Layer 2 is an optional enterprise extension for complex interactions.

Here is a guide to choosing the correct level of abstraction based on your needs:

When to use LAYER 1 (Core Inventory System)

Layer 1 should be used for everything related to the internal logic and structure of an inventory belonging to a single Actor (such as the Player Character, an NPC, or a container in the world).

Choose the Layer 1 APIs if you need to manage:

  • Internal structure: Define whether an inventory uses simple slots, a rotatable NxM grid, or is composed of typed equipment slots (Head, Torso, etc.).
  • Nested Objects (Nesting): Manage the logic of “objects inside other objects,” such as a bag containing ammunition or a weapon module inserted into a rifle.
  • Local Operations: Add, remove, move, or split items within the same component.
  • Single Player or NPC: Manage a character’s equipment or the contents of a chest when there is no simultaneous interaction between multiple players that requires locking systems.
  • Base Backend: Layer 1 provides the Flat Registry, Delta Replication, and the fundamental Prediction to eliminate latency on the single component.

When to use LAYER 2 (Cross-Inventory Interaction)

Layer 2 should be used exclusively when an operation involves two distinct inventories (Endpoints) and requires enterprise-grade security and atomicity guarantees.

Choose the Layer 2 APIs if you need to manage:

  • Player-World Interaction: Typical scenarios such as Player <-> Chest, access to personal Stashes, looting bodies or lockers.
  • Locking System (Multi-view / Single-writer): If you want multiple players to be able to look simultaneously inside the same chest, but only one at a time can move items to avoid conflicts or duplication.
  • Atomic Cross-Component Transactions: Move an entire “sub-tree” of objects (e.g., a full bag) from one inventory to another with the guarantee that the action happens fully on both sides or does not happen at all (Rollback Journal).
  • Security and Capability: Define granular permissions (e.g., “this player can only view but not withdraw”) based on a server-authorized Session.
  • Proximity Gating: When you want the connection between player and chest to be automatically suspended if the player moves too far away (Suspended State).

Summary for Decision Making

Scenario Recommended Layer Why?
Moving an item within the backpack Layer 1 It is an internal mutation within the same component
Equipping a weapon Layer 1 It concerns the slot logic of a single character
Taking gold from a chest Layer 2 It involves two owners and requires a session protocol
Placing a bag into a chest Layer 2 Requires a Subtree Transfer between two distinct endpoints
Adding ammo to an NPC Layer 1 Local management of the NPC's records
Opening a loot box with other players Layer 2 Handles concurrency via the Writer Lock
💡 Summary

In short: Layer 1 manages the inventory “itself,” while Layer 2 manages the “secure bridge” between two inventories. Layer 2 always depends on the foundations of Layer 1, but Layer 1 can be used on its own for projects that do not require complex cross-inventory interactions.
