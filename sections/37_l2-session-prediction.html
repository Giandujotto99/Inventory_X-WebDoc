Session Prediction

Session Prediction is the Layer 2 layer designed to eliminate perceived latency during operations involving two distinct inventories (for example, between the player and a chest).

Unlike single-component simulation, this architecture simultaneously coordinates changes on both endpoints of the session.

Unified Endpoint Coordination

The system does not simply add two independent predictions together, but uses a dedicated Session Prediction Manager that acts as the single arbiter of local consistency.

Simultaneous Overlay: When the player moves an item, the system instantly applies a visual overlay to both the source inventory and the destination inventory.

Shared Transactional Model: The client uses the same planning engine as the server to build the local “transaction plan,” drastically reducing the risk of semantic divergence.

Identity Management (Scoped Remap)

To handle complex operations, such as moving backpacks or splitting stacks that create new entities, the system implements a Scoped Remap:

The remap tracks and corrects references not only for individual items, but also for child containers created dynamically during the simulation.

This ensures that subsequent operations (e.g., putting an item into a bag that was just moved) remain valid even if the server assigns final identifiers different from those predicted by the client.

Barrier-Driven Reconciliation

The final confirmation of an action does not depend on the simple arrival of the server response (Ack), but on the fulfillment of a Review Barrier:

The operation is considered complete only when the client has received and applied the network updates for both inventories involved.

This prevents inconsistent states where an item appears removed from one UI panel but has not yet appeared in the other due to replication delays in one of the two components.

Session Security and Integrity

The prediction model is designed to fail safely in the presence of instability:

Automatic Invalidation: Any change in the logical boundary of the session (such as switching to the Suspended state or a SessionEpoch change) causes the immediate clearing of the local prediction queue.

Local Prevalidation: The client performs a preliminary check based on its actual capabilities (e.g., “can withdraw,” “can deposit”); if the action is not authorized locally, the request is not even sent to the server, saving bandwidth.

Efficiency and Timeouts

To maintain high performance, the system avoids continuous monitoring processes (per-frame ticks). Timeout management for pending operations is handled by discrete timers that intervene only when necessary to force realignment with the server’s authoritative state in case of network issues.
