Capability Model

The Capability Model is the granular authorization system of Inventory_X’s Layer 2, designed to define with surgical precision what a player can or cannot do within a specific interaction session,.

Instead of using generic permissions, the system adopts a server-authoritative approach based on explicit and dynamic rights, ensuring maximum security in multiplayer contexts,.

Here are the fundamental pillars of this model:

1. The Capability Snapshot (Authoritative Snapshot)

The heart of the system is the Capability Snapshot, a “snapshot” of the actual rights granted by the server for a given epoch of the session,.

Immutability per Epoch: Once generated, the snapshot is valid only for the current SessionEpoch,. If conditions change (e.g., the player moves away or loses a permission), the server increments the epoch and generates a new snapshot, instantly invalidating operations based on old permissions,.

Protected Replication: The snapshot is replicated exclusively to the session owner (Owner-Only), preventing other clients from spying on others’ permissions,.

2. Granularity of Actions

The model is not limited to simple “Read Only” or “Write” access, but breaks interaction down into atomic capabilities. Among the main ones are:

Inspection: Ability to see the contents of the external inventory.

Transfer: Distinct rights for depositing (Deposit) or withdrawing (Withdraw) items,.

Advanced Manipulation: Specific permissions to split (Split), merge (Merge), or swap (Swap) items between the two inventories,.

Session Management: The ability to request a privilege increase (Upgrade) or the restoration of a suspended session,.

3. State-Based Permission Matrix (Least Privilege)

The system strictly applies the principle of least privilege (Least Privilege), varying the snapshot based on the session lifecycle state:

Active State (Read Only): Only inspection capabilities and the ability to request write access are granted,.

Active State (Exclusive Write): Only when the session holds the Writer Lock are mutative capabilities (move, merge, etc.) unlocked,.

Suspended State: In this security state, almost all capabilities are revoked, leaving only the right to request session Resume active,.

4. Dynamic Derivation and Security

The Snapshot is not static data provided by the client, but is calculated exclusively by the server by cross-referencing multiple sources of truth,:

Provider Policy: The rules defined in the external inventory Data Asset (e.g., “this locker does not allow deposits”),.

Requester Context: The trusted identity of the player (faction, game permissions, equipped items),.

World Conditions: Verification of physical distance and line of sight relative to the container,.

5. Usage in UI and Prediction

Although authority belongs to the server, the Capability Snapshot is used by the client for Local Prevalidation. This allows the user interface to show immediate feedback (e.g., disabling the “Withdraw” button if the capability is not present in the current snapshot) and local prediction to block actions from the start that the server would certainly reject, saving bandwidth and resources,.
