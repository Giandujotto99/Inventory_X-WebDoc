Session Delegates and Events

The Layer 2 Delegate and Event system is the reactive architecture that allows the UI and game systems to respond to changes in the session lifecycle and to the outcome of cross-inventory operations.

All delegates are defined as UPROPERTY(BlueprintAssignable) within the UInventoryInteractionComponent, ensuring full compatibility with Blueprints.

1. Lifecycle Events (Session Lifecycle)

These events notify structural changes in the link between the player and the external inventory:

OnInteractionSessionOpened: Emitted when the session is successfully activated (in read-only or write mode).

OnInteractionSessionClosed: Notifies the definitive closure of the session and the release of all local resources.

OnInteractionSessionSuspended: Triggers when a suspension cause occurs (e.g., loss of range or visibility), temporarily invalidating active operations.

OnInteractionSessionResumed: Confirms that the session has become operational again after a suspension.

OnInteractionLockStateChanged: Fundamental for the multi-view model; it is emitted not only when the current player acquires or releases the Writer Lock, but also when their session observes a change of lock ownership by another user, allowing the UI to update edit permissions in real time.

2. Operation Events (Prediction-Aware)

These delegates follow the barrier-driven model of phase 8, ensuring that player feedback is always synchronized with the authoritative truth of the inventories:

OnInteractionOperationPredicted: Emitted instantly when a local simulation has been successfully applied, allowing the UI to immediately show the item in the new inventory.

OnInteractionOperationConfirmed: The operation is final. It is emitted only after the RevisionBarrier has been satisfied on both endpoints and the server result is semantically equivalent to the prediction.

OnInteractionOperationRejected: The action was rejected by the server or expired due to timeout; the system automatically initiates rollback of the local state.

OnInteractionOperationCorrected: The barrier is satisfied, but the real result differs from the simulation (e.g., different position); the UI must update to reflect the real server data.

OnInteractionOperationInvalidated: Triggers if a pending operation is canceled due to a session boundary change (e.g., sudden suspension) or due to a cascading failure during replay.

3. Delivery and Integrity Rules

To ensure maximum enterprise stability, Layer 2 follows strict emission rules:

Unique Outcome: For each OperationId, only one final closure can exist among Confirmed, Rejected, Corrected, or Invalidated.

Cross-Role Consistency: The system uses unified emission helpers so that delegates behave identically in single-player, on remote clients, or on a listen server.

No Polling: The architecture is entirely event-driven; widgets must redraw only in response to these signals, drastically optimizing performance.
