Architectural Principles

The architecture of Inventory_X is built on foundations designed to ensure scalability, multiplayer security, and maximum performance. The system follows seven key principles.

1. Backend-First and UI-Agnostic

The core of the plugin is focused exclusively on data, rules, operations, and networking. The system does not impose widgets, graphical layouts, or input bindings. This separation allows developers to build completely customized user interfaces (from simple lists to complex grids) that consume the plugin APIs without dependencies on rendering.

2. Server-Authoritative with Client Prediction

The source of truth is always the server. To eliminate perceived latency, the client locally simulates operations (Prediction), providing immediate feedback to the UI. However, every action is validated and confirmed by the server; in case of discrepancies, the system automatically handles reconciliation (Reconciliation) to realign the client with the authoritative state.

3. Trusted Identity (Authority Context)

Security is not based on data sent by the client. The server independently reconstructs an Authority Context (FInventoryAuthorityContext) based on the real network connection. This ensures that access policies (such as “Owner Only”) are verified against the real identity of the requester and not against tags declared by the client that could be forged.

4. Data-Driven and Tag-Driven

The system’s behavior is not hardcoded. Stacking rules, slot compatibility, and placement restrictions are defined through reusable Data Assets and GameplayTags. This allows designers to modify gameplay dynamics (e.g., create a new category of weapons or containers) without programmer intervention.

5. Lightweight Structures and Flat Registry

To optimize memory and networking, objects are not modeled as heavy UObject instances per item. The system uses flat data structures (Flat Registry) composed of USTRUCTs and referenced through stable Handles. This approach drastically simplifies network delta replication (FFastArraySerializer) and significantly improves performance in inventories with hundreds of items.

6. Layered Architecture (Layering)

The plugin is divided into independent modules to manage complexity in a granular way:

  • Layer 1 (Core): Manages the fundamental logic of a single component (grid, nesting, internal rules).
  • Layer 2 (Interaction): Coordinates secure transactions between two distinct components (e.g., Player <-> Chest) by introducing sessions and locking systems.

7. Operation Atomicity

Every state change follows a rigorous pipeline: Validate -> Simulate -> Commit. Complex operations, such as swapping two items (Swap) or transferring a full bag (Subtree Transfer), are treated as atomic transactions: either the entire operation succeeds on both endpoints, or no change is applied, ensuring data integrity at all times.
