Glossary

To fully understand the architecture of Inventory_X, it is essential to become familiar with the key terms that define the data model and the operational flow of the system.

Core Concepts (Layer 1)

  • Definition: A static and reusable asset (UDataAsset) that describes the immutable properties of an item or a container, such as dimensions, stacking rules, and cosmetic data.
  • Record: The real instance of an item or container present in memory during gameplay. Records are stored efficiently in "flat" arrays (Registry Flat) to optimize performance and network replication.
  • Handle: A stable, lightweight, and safe reference (composed of an Id and a Generation number) used to point to a runtime record. Handles avoid the use of direct pointers that could become invalid if memory is reallocated.
  • Fragment: An optional dynamic data block associated with an item, used to manage properties that change over time, such as weapon durability or the number of bullets in a magazine.
  • View: A read-only snapshot of the data of an item or container. Views are designed to be safely passed to the UI or to Blueprints without risking unintentional modification of the internal inventory state.
  • Operation: An atomic request to modify the state of the inventory (e.g., moving an item, equipping a weapon). Each operation follows a strict pipeline of validation, simulation, and confirmation from the server.

Enterprise Concepts (Layer 2)

  • Interaction Session: An authorized and server-controlled link between a player's inventory and an external inventory (e.g., a chest).
  • Endpoint (Primary & Secondary): In an interaction session, the Primary endpoint is the player's inventory, while the Secondary endpoint is the external inventory being interacted with.
  • SessionEpoch: An identifier of the session’s "logical generation". The value increases every time the security context changes (e.g., when write permission is gained or lost), invalidating pending operations from the previous generation.
  • Writer Lock: The exclusive right to modify an external inventory. While many players can view the same chest simultaneously, only one at a time can hold the Writer Lock to move or add items.
  • Capability Snapshot: A snapshot of the effective permissions granted by the server to the current session, which determines which actions (view, deposit, withdraw, etc.) are allowed at that moment.
  • Scoped Handle: A handle that explicitly specifies which endpoint it belongs to (whether the player or the external container), allowing item identities to be managed correctly across different components.
  • Subtree Snapshot: A complete representation of an item and all of its recursive contents (e.g., a bag full of items), used to transfer complex hierarchies between two distinct inventories.
  • Revision Barrier: The synchronization point that the client must observe on both endpoints to consider a cross-inventory operation as definitively confirmed by the server.
  • Suspended: A transitional state in which the session remains open but "frozen" (e.g., because the player moved too far away from the chest), preventing new operations until a restore (Resume) is performed.
