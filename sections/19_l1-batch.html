Batch Operations

Batch Operations represent an advanced feature of the Layer 1 of Inventory_X, designed to handle scenarios where it is necessary to execute multiple related inventory changes as if they were a single indivisible action.

This system is ideal for massive operations such as completely emptying an inventory, loading a predefined loadout, or simultaneously collecting multiple items from a container in the world.

Here are the fundamental details about how the batch system works and its constraints in the current version:

1. "All-or-Nothing" Semantics

The core principle of batch operations is atomicity. When an FInventoryBatchRequest is submitted, the system validates and simulates the entire sequence of operations before applying any change.

If even a single operation within the batch fails (due to space, tag, or permission rules), the entire batch is rejected and the inventory remains exactly in the state it was in before the request.

2. Non-Cumulative Model (v1)

A specific architectural choice of version v1 is that the batch is non-cumulative.

Each individual operation within the group is validated against the state of the inventory prior to the batch, and not against the results of the operations that precede it in the list.

Example: If operation A frees a slot and operation B attempts to occupy that same slot in the same batch, operation B may fail because, at the time of validation, the slot still appears occupied by the item removed in A.

This limitation was introduced to guarantee maximum performance and reduce the complexity of calculating Change Sets, while still covering the vast majority of real-world use cases (independent operations on different items).

3. Technical Limits and Constraints

To ensure system stability and prevent abuse or network overload, the following constraints are imposed:

Limit of 32 Operations: Each batch can contain a maximum of 32 individual operations (MaxBatchOperations). Any request that exceeds this limit is rejected with the error code BatchLimitExceeded.

Single Component: The operations in a batch must all belong to the same UInventoryComponent. It is not possible to include changes that span different components in a single Layer 1 batch.

No Conditional Logic: The batch is an ordered and fixed list; it does not support "if... then" instructions or alternatives in case a sub-operation fails.

4. Notifications and Aggregated Events

To optimize UI performance, the system does not emit an event for each individual change contained in the batch. At the end of the atomic commit of all operations, a single aggregated OnInventoryChanged event is emitted. This prevents the UI from performing unnecessary and costly multiple refreshes within the same frame.

5. Networking and Prediction

In multiplayer, batch operations follow the same strict security protocol as single operations:

Atomic Prediction: The client simulates the entire batch instantly, providing smooth feedback to the player.

Single Reconciliation: The server validates the batch and responds with a single confirmation or rejection message (Ack), ensuring that client and server remain perfectly synchronized over the entire block of changes.
